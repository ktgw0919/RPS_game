# Product Requirements Document (PRD)

## 1. プロジェクトの目的と概要
対面またはオンラインで集まった複数人が、各自のスマートフォンやPCのブラウザを利用して参加できる「リアルタイムじゃんけんゲーム」を開発する。
フルスタック開発（React + FastAPI + WebSocket + MongoDB）の学習と実践を兼ねており、単純なじゃんけんだけでなく、システム側で複雑な勝敗判定を行う特殊ルールを実装することで、状態管理や非同期処理の技術検証を行う。

## 2. ターゲットユーザーと利用シナリオ
- **ターゲット:** 複数人のグループ（3人〜程度を想定）。
- **利用シナリオ:** 1. 代表者（ホスト）がスマホでブラウザを開き、「ルーム」を作成する。
  2. ルームID（または共有リンク／QRコード）を他の参加者（ゲスト）に共有する。※ 共有リンク・コードコピーは実装済み。QR 表示は未実装。
  3. ゲストはアプリのインストール不要で、ブラウザから名前を入力してルームに参加する。
  4. ホストがゲームを開始し、全員が手元のスマホで手（グー・チョキ・パー）を選択する。
  5. 全員の画面にリアルタイムで結果と勝敗が表示される。

## 3. 機能要件 (Functional Requirements)

### MVP (Minimum Viable Product)（実装フェーズ: Phase 1–2・4）
まずは最もシンプルな「一斉じゃんけん」をオンラインで完結させる。
- [x] **ルーム管理機能**
  - ホストは一意のルームコードを生成できる（4桁の大文字英数字。`0 O 1 I L` など紛らわしい文字は除外し、衝突時は再生成）。
  - ゲストはルームコードと表示名を入力して参加できる。1ルームの上限は20人とする（定員は接続中/切断中の人間・観戦者・CPU の合算で数える。詳細は `ARCHITECTURE.md` §3.1）。表示名はトリム後1〜20文字・制御文字禁止（絵文字/多言語可・同名許容。詳細は `ARCHITECTURE.md` §3.2）。
  - 最終操作から30分間無操作のルームは自動で破棄する。マッチ終了後やロビーで一定時間復帰しない切断プレイヤー（ゴースト）は除去して定員枠を解放する（詳細は `ARCHITECTURE.md` §6/§10）。
  - ホストが切断したまま復帰しない場合は、最古参の接続中プレイヤーへホストを自動移譲する。
  - パスワード等の複雑な認証は行わない。代わりに、ルーム作成/参加時にサーバーが `playerId` と推測困難な `playerToken` を発行し、クライアントは Local Storage（または Cookie）に保持してユーザーを識別する。
  - WebSocket 接続時にこのトークンを提示し、**再接続時は同一トークンで元のプレイヤーに再紐付け**する（新規プレイヤーとして二重登録しない）。各操作はトークンに紐づく本人のものとして検証する。
- [x] **リアルタイム同期機能 (ロビー)**
  - ルーム内の参加者一覧がリアルタイムで全員の画面に同期される。
- [x] **基本ゲームループ機能（オンライン判定モード）**
  - ホストが「ゲーム開始」をトリガーする。
  - 参加者は制限時間内に手（グー・チョキ・パー）を選択して送信する（締切前は上書き可）。
  - 生存者全員の手が揃った時点、または**サーバー側タイマーの締切到達**時に、サーバーが勝敗を判定し、全員に結果をブロードキャストする。締切までに未提出のプレイヤーは**そのラウンド敗北（脱落系ルールでは脱落）**として扱う。ただし**提出者が0人／判定で生存者が0人**になる場合は、その判定を無効化して同メンバーで再ラウンドする（あいこ扱い・あいこ上限到達で引き分け終了。詳細は `ARCHITECTURE.md` §7）。
  - 判定で「あいこ」になった場合は再ラウンドを行う（あいこ回数の上限はホスト設定。上限到達時は引き分け終了）。
- [x] **ホスト設定機能**
  - ホストはロビーで次の項目を設定でき、変更は全員のロビー画面にリアルタイム反映される。
    - ルール種別（特殊ルールは Phase 3 で有効化）／NORMAL のマッチ終了方式（脱落式・1ラウンド確定）／ラウンド制限時間（秒）／次ラウンド進行モード（自動・手動の切り替え）／1マッチ内のあいこ回数上限
  - 進行モードが「手動」の場合、ホストの操作で次ラウンド・次フェーズへ進める。
- [x] **途中参加・再接続の扱い**
  - ゲーム進行中の新規参加者は観戦として入室し、次のマッチからプレイ参加する。
  - 通信が切断されたプレイヤーは、そのマッチ終了まで生存状態を維持し、同一トークンでの再接続により状態を復元して復帰できる。マッチ進行中の意図的な退室（`LEAVE`）も切断と同等に扱い、そのマッチ終了まで生存を維持する（ロビー／マッチ終了時の退室は即時）。※ `LEAVE` は WS で実装済みだが、フロントの退室ボタンは未実装。
- [x] **開発/デモ用 CPU プレイヤー（開発・デモ補助機能）**
  - 一人でも動作確認・デモができるよう、ホストはロビーで **CPU プレイヤーを追加/削除**できる（実体のないサーバー制御プレイヤー。WebSocket 接続・トークンを持たない）。
  - CPU は通常プレイヤーと同様に**定員（20人）・開始最小人数にカウント**する（ソロ＋CPU1体で NORMAL を開始できる）。
  - CPU は毎ラウンド**サーバーが自動で手を出す**（MVP は完全ランダム。戦略は後フェーズで拡張）。タイムアウト・切断・退室は発生せず、ホストにもならない。
  - この機能は **`ALLOW_CPU`（`.env`）で切替**でき、**本番では無効化を推奨**する（開発/デモ用途のため。詳細は `ARCHITECTURE.md` §3/§5/§11）。

### 特殊ルールの実装（実装フェーズ: Phase 3）
MVP完成後、以下の特殊ルール（モード）を実装する。
- [ ] **少数派勝利ルール (Minority Rule)**
  - 最も人数の少なかった手を出したプレイヤーのみが生存する。
  - 全員同じ手・最少人数の手が複数タイの場合（あいこ）の再試合ロジックをシステムで自動制御する。
  - **生存者が閾値以下（例: 2人以下）になったら通常じゃんけん（NORMAL）で決着する**。閾値・移行タイミング（即時／次マッチから等）はホストが設定できる。
- [ ] **代表ルール (Boss Battle)**
  - 1人の「代表（ボス）」とN人の「参加者」に分かれる非対称モード。ボスはホストが指名する。
  - **ボスは非参加者扱いとし、勝者としてはカウントしない**（毎ラウンド手は出す）。ボスに勝った参加者のみが生き残る（またはスコアを獲得する）。
- [ ] **1対1トーナメント自動進行**
  - 参加者から自動でトーナメント表を生成し（奇数は不戦勝を割り当て）、ペアごとに独立した判定を行う。ペア内のあいこはそのペアのみ再戦する。

## 4. 非機能要件 (Non-Functional Requirements)
- **モバイルファーストのUI/UX:** 参加者は主にスマートフォンで操作することを前提とし、タップしやすい大きなボタンと視認性の高いレイアウト（Tailwind CSS）を採用する。
- **リアルタイム性:** WebSocketを利用し、手の入力から結果表示までの遅延を最小限に抑える。
- **耐障害性:** ラウンドはサーバー側タイマーを権威とし、締切までに未提出のプレイヤーは「敗北/脱落」として処理してゲームの進行がストップしないようにする。通信が切断されたプレイヤーはそのマッチ終了まで生存を維持し、再接続で復帰できる。
- **データ保存:** 対戦履歴・スコアを MongoDB に永続化する。**MVP ではルーム単位のマッチ履歴のみ**を保存し（`room_code` で参照）、`display_name` 横断のグローバルなスコアボードは後フェーズ（アカウント連携時）に設計する。※ `match_history` への書き込みは未実装（DB 接続のみ。`docs/TODO.md`「MVP 残タスク」参照）。
- **開発/デモ容易性:** 一人でも全ゲームループを検証できるよう CPU プレイヤーを用意する。CPU は `ALLOW_CPU`（`.env`）で有効/無効を切り替えられ、本番では無効化を推奨する。

## 5. 制約事項 (Constraints)
- バックエンドのPython環境・依存管理には `uv`（`pyproject.toml`）を使用する。`requirements.txt`(pip) は使用しない。
- バックエンドのスキーマ定義は **Pydantic v2** を採用し、アプリのライフサイクルは `lifespan` で管理する（`@app.on_event` は使用しない）。
- 設定値は `python-decouple` を用いて `.env` から読み込む（`DB_URL`, `DB_NAME` は必須。`REDIS_URL`, `ALLOW_CPU` は任意）。`ALLOW_CPU` は開発/デモ用 CPU プレイヤーの有効化フラグ（本番は無効化推奨）。`CORS_ORIGINS` と運用調整値（`HOST_TRANSFER_GRACE_SEC` / `GHOST_TTL_SEC` / `ROOM_IDLE_TTL_SEC` / `ROOM_CREATE_RATE_MAX` / `ROOM_CREATE_RATE_WINDOW_SEC`）も任意（既定値つき・fail-fast 対象外）で `.env` 上書きできる。一方、ハートビート間隔・WS メッセージサイズ上限・CPU 提出遅延・スイープ実行間隔などプロトコル/体感の固定値は `.env` ではなくコード定数（`core/constants.py`）に置く（詳細は `ARCHITECTURE.md` §1/§4/§6/§10/§11）。
- **ゲーム進行状態の正本は、MVP では単一プロセスのインメモリとする**。状態アクセスはインターフェース越しに抽象化し、Redis は水平スケール時に差し替え可能な任意要素とする（MVP は単一ワーカー前提）。永続化が必要な対戦履歴・スコアのみ MongoDB に保存する。
- UIフレームワークには React を採用し、**Vite + TypeScript（strict）** で SPA（Single Page Application）として構築する（Create React App は使用しない）。フロントのゲーム状態は React Context + `useReducer` で管理し、REST 取得用の SWR は MVP では導入しない（履歴・スコアボード実装時に導入）。
- **MVP の運用前提**: ゲーム進行はインメモリ正本のため uvicorn は単一ワーカーで起動し、**プロセス再起動で進行中のゲーム状態は失われる**ことを許容する（永続化は `match_history` の確定結果のみ）。
- **品質ツール**: backend は ruff（lint+format）・mypy（型）・pytest、frontend は ESLint + Prettier・TypeScript strict を用い、**GitHub Actions の CI** で lint・型チェック・テストを実行する。
- **ローカル MongoDB** は `docker-compose` 方式と MongoDB Atlas 方式の両方を README に併記し、`.env`（`.env.example` 同梱）で切り替える。
- ディレクトリ構成・責務分割は `ARCHITECTURE.md` §2 を正とする。雛形メモ（`docs/TEMPLATE_NOTES.md`）の `Layout`/`components` 構成・シードスクリプト等は踏襲しつつ、backend の責務は `routers/`（I/O 境界）/ `core/`（接続・状態・セキュリティ基盤）/ `game/`（純粋なゲームドメイン）の3層に分割する（汎用ヘルパのみ `utils.py`）。
  - ただし雛形は REST のみで WebSocket を含まないため、リアルタイム層（WebSocket ルーター・`ConnectionManager`・判定エンジン）は本プロジェクトで新規に追加する。
- 将来的には別システムのアカウントシステムとの連携を考慮する。
